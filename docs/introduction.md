# CloudFormation ワークショップ 導入

## Infrastructure as Code（IaC）とは

Infrastructure as Code（IaC）とは、サーバーやネットワークなどのインフラを**コード（テキストファイル）で定義・管理する**手法です。

AWS では従来、マネジメントコンソール（GUI）や AWS CLI でリソースを 1 つずつ手作業で作成してきました。IaC ではその代わりに、作りたい構成を**テキストファイルに記述**し、ツールに実行させます。

---

## GUI / CLI との比較

### 手作業の課題

VPC → サブネット → セキュリティグループ → EC2 → ALB を手作業で構築する場面を想像してください。

| 手作業 | IaC |
|---|---|
| コンソールを何画面も行き来して設定する | テキストファイル 1 つに全リソースを記述する |
| 「あの設定どこで変えたっけ」と迷う | ファイルを開けば全設定が見える |
| 同じ環境をもう 1 つ作るのが大変 | 同じファイルで何度でも作れる |
| 誰がいつ何を変えたか追えない | Git で変更履歴が残る |
| 手順書と実際の環境がずれていく | コードが常に環境の実態と一致する |

### IaC で変わること

**再現性** — 同じテンプレートから同じ環境を何度でも作れる。開発・ステージング・本番を同じ定義から構築できる。

**変更の追跡** — Git でテンプレートを管理すると「いつ・誰が・何を・なぜ変えたか」が履歴に残る。GUI での変更は記録が残らない。

**レビュー** — インフラの変更をコードの差分として確認でき、チームでレビューしてからデプロイできる。

**一括管理** — 10 個のリソースをまとめて作成・更新・削除できる。手作業だと 1 つずつ操作して、順番も自分で考える必要がある。

---

## CloudFormation とは

AWS CloudFormation は、AWS が提供する IaC サービスです。

- **テンプレート**（YAML / JSON）に作りたい AWS リソースを定義する
- テンプレートを CloudFormation に渡すと、記述通りのリソースが自動で作成される
- テンプレートを変更して再度渡すと、**差分だけ**が適用される
- テンプレートから作られたリソースの集まりを**スタック**と呼ぶ

```
テンプレート（YAML ファイル）
       │
       ▼
  CloudFormation
       │
       ▼
  スタック（AWS リソースの集まり）
  ├── VPC
  ├── サブネット
  ├── セキュリティグループ
  └── EC2 インスタンス
```

## テンプレート = パラメータシート

CloudFormation のテンプレートは最初は難解に感じるかもしれません。しかし、皆さんには馴染みのある概念に置き換えられます。

### システム構築のパラメータシートとの類似

サーバーやネットワークを構築するとき、Excel などで**パラメータシート**を作った経験があるはずです。

```
【パラメータシート例】

■ VPC
  ・名前: workshop-vpc
  ・CIDR: 10.0.0.0/16
  ・DNS サポート: 有効
  ・DNS ホスト名: 有効

■ サブネット1
  ・名前: workshop-public-1a
  ・VPC: (上で作った VPC)
  ・CIDR: 10.0.1.0/24
  ・AZ: ap-northeast-1a
  ・パブリック IP 自動割当: 有効
```

これを CloudFormation テンプレートで書くと、こうなります。

```yaml
Resources:
  VPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: "10.0.0.0/16"
      EnableDnsSupport: true
      EnableDnsHostnames: true
      Tags:
        - Key: Name
          Value: "workshop-vpc"

  PublicSubnet1:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: "10.0.1.0/24"
      AvailabilityZone: "ap-northeast-1a"
      MapPublicIpOnLaunch: true
      Tags:
        - Key: Name
          Value: "workshop-public-1a"
```

**パラメータシートと CloudFormation テンプレートは同じ情報**を持っています。違いは以下の通りです。

| パラメータシート | CloudFormation テンプレート |
|---|---|
| 人間が読んで手作業で設定する | CloudFormation が読んで自動で設定する |
| 「(上で作った VPC)」のような曖昧な参照 | `!Ref VPC` で明確に参照する |
| シートを更新しても環境には反映されない | テンプレートを更新すると環境に反映される |
| シートと環境がずれる（ドリフト） | テンプレート = 環境の実態 |

つまり CloudFormation テンプレートは、**機械が読めるパラメータシート**です。書いてある情報はこれまでと同じ。ただし人間の代わりに CloudFormation が読んで実行してくれるため、設定ミスや手順漏れがなくなります。

## このワークショップでやること

このワークショップでは、最小構成のテンプレートから始めて段階的にリソースを追加し、テンプレートを育てながら以下を学びます。

1. **テンプレートの書き方** — リソース定義、パラメータ、関数、条件分岐
2. **スタック操作** — 作成、更新、クロススタック参照、削除
3. **完成形** — VPC + ALB + Auto Scaling + EC2 でカレンダーアプリをホスト

```
Step 1   VPC だけ
Step 2   + サブネット
Step 3   + IGW + ルートテーブル
Step 4   + セキュリティグループ + Outputs
            ── ネットワークスタック完成 ──
Step 5   EC2 1 台（新しいスタック）
Step 6   + ALB
Step 7   + Launch Template + Auto Scaling + Conditions
            ── アプリケーションスタック完成 ──
```

各ステップで新しいテンプレート要素や関数が登場しますが、一度に全部覚える必要はありません。パラメータシートに 1 行ずつ設定値を追記していくのと同じ感覚で、テンプレートにリソースを 1 つずつ追加していきます。
